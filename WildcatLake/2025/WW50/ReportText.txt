  Revert clearing DISB when ME code corruption is detected
  ^^^^^^                                                   
  
  [Issue Description]
  The function PlatformInitRecoveryEntryPoint()
  clears DISB when ME code corruption is detected to
  mitigate MRC PC-DF55 hanging issue.
  However when attaching a WLAN card which requests a different
  WLAN Microcode in FW also makes ME code corruption detection
  stand due to Microcode mismatch.
  Clearing DISB will break Capsule update flow since Capsule
  update requires Capsule image to be intact in system memory.
  So if system is booting with WLAN Microcode mismatch, Capsule
  update can never be executed in BIOS.
  
  [Resolution]
  After revisiting the PC-DF55 issue, it's relative to TCO
  timeout GBLRST in Resiliency debug BIOS: Resiliency debug BIOS
  uses release build FSP to accommodate IBB 4MB space, however
  release build FSP does not halt TCO timer in earlier phase plus
  the platform code to halt TCO timer is too late.
  
  So the solution is halting TCO timer in earlier phase to resolve
  ME corruption DF55 issue, and thus the mitigation of clearing
  DISB when detecting ME code corruption can be removed and
  Capsule update wont be broken.
